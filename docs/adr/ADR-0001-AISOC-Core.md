ADR-0001: AISOC Core
AI Security Operations Center как архитектурное ядро агентной системы
Статус: Accepted
Дата: 29.12.2025
Контекст: Agent-based AI System / Multi-Agent Platform
Владелец решения: Architecture / Security
________________________________________
1. Контекст и проблема
Система использует ИИ и агентную логику для взаимодействия с корпоративными системами и внешней средой.
Агенты по своей природе:
•	не являются субъектами ответственности;
•	могут быть скомпрометированы через prompt-инъекции и логические атаки;
•	принимают решения в runtime, что делает post-factum контроль недостаточным.
Проблема:
Без централизованного контрольного слоя система:
•	не соответствует требованиям ИБ и регуляторов;
•	не поддаётся аудиту и воспроизводимости;
•	не масштабируется за пределы демо / PoC.
________________________________________
2. Принятое архитектурное решение
В архитектуру системы вводится AISOC (AI Security Operations Center)
как обязательное архитектурное ядро (control plane) для всех агентных взаимодействий.
AISOC является единой точкой контроля, через которую проходят:
•	действия агентов;
•	доступ к данным;
•	обращения к внешним и внутренним сервисам.
________________________________________
3. Границы решения (Scope)
AISOC отвечает за:
•	регистрацию и идентификацию агентов;
•	уровни доверия и разрешённые действия;
•	принудительное исполнение политик доступа;
•	runtime-проверку всех запросов агента;
•	аудит и explainability решений;
•	реагирование на инциденты (block / degrade / escalate / kill-switch);
•	обучение защитных механизмов по модели антивируса;
•	симуляцию угроз (red-team, Top-100 сценариев).
AISOC НЕ отвечает за:
•	бизнес-логику агентов;
•	обучение предметной области;
•	формирование политик доступа;
•	принятие бизнес-решений.
________________________________________
4. Взаимодействие с другими контурами
Контур	Роль
Decision / Solution Layer	Принимает бизнес-решения под контролем AISOC
Audit Layer	Обеспечивает неизменяемый аудит и воспроизводимость
Risk Analyzer	Поставляет сигналы риска, но не управляет доступом
AITeacher / Trainer	Обучает защиту, не меняя доступы автоматически
Connectors / Adapters	Реализуют доступ к системам только через AISOC
AISOC — не модуль, а архитектурный центр системы.
________________________________________
5. Принцип применения политик
•	Политики задаются на уровне организации, а не ИИ.
•	Данные:
o	предварительно подготовлены;
o	классифицированы;
o	привязаны к уровням доступа.
•	AISOC исполняет политики, не интерпретируя намерения агента.
________________________________________
6. Модель обучаемости (Security Evolution)
AISOC использует модель:
White List (статический контур)
+
Threat Intelligence (динамический контур)
Разрешено автоматически:
•	детектировать атаки;
•	усиливать мониторинг;
•	блокировать и деградировать ответы;
•	обновлять сигнатуры угроз.
Запрещено автоматически:
•	расширять доступ;
•	снимать ограничения;
•	изменять уровни доверия.
Все изменения проходят цикл:
Detect → Analyze → Sandbox → Approve → Deploy → Audit
________________________________________
7. Уровни зрелости (reference)
•	L0 — Demo / Sandbox (без AISOC, только демонстрация)
•	L1 — Controlled MVP (ограниченный контроль)
•	L2 — AISOC Core ✅ (минимум для production)
•	L3 — Adaptive Security
•	L4 — Enterprise / Regulated
AISOC Core является точкой перехода от демо к продукту.
________________________________________
8. Нормативное соответствие (reference)
Архитектура согласуется с принципами:
•	ISO/IEC 27001 / 27002
•	NIST AI RMF
•	ISO/IEC 23894
•	GDPR / EU AI Act (для EU-зон)
•	SOC 2 / NIST (для US-зон)
________________________________________
9. Последствия принятого решения
Плюсы:
•	система становится auditable и explainable;
•	снимается блокировка со стороны ИБ и compliance;
•	появляется единая точка контроля;
•	масштабирование агентов не увеличивает риск.
Минусы:
•	увеличение сложности архитектуры;
•	необходимость явного проектирования политик;
•	невозможность «быстрого хака» без контроля.
________________________________________
10. Итоговое решение
AISOC Core является обязательным архитектурным элементом.
Без него система остаётся демо.
С ним — становится продуктом.
________________________________________
Готово к использованию как:
•	ADR в репозитории;

•	аргумент для ИБ / заказчика / инвестора.
